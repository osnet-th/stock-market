# 월급 사용 비율 (Salary Usage Ratio) — 브레인스토밍

**날짜**: 2026-04-16
**관련 이슈**: #26
**상태**: 초안 (승인 대기)

---

## What We're Building

"월급 사용 비율"이라는 **신규 탭**을 추가하여, 사용자가 월 실수령액과 카테고리별 사용 내역을 입력하면 다음을 시각화한다.

### 핵심 시각화
1. **도넛 차트**: 월급 대비 카테고리별 사용 비율 (%) — "미사용/잔여" 세그먼트 포함
2. **바 차트**: 카테고리별 사용 금액 (원)
3. **라인 차트 (최근 12개월 추이)**: 총 지출 / 월급 / 저축율 3개 라인

### 데이터 입력 모델 — **변경 지점 기반(Effective Date)**
- 사용자는 **바뀌는 달만 값을 입력**한다.
- 입력된 값은 **다음 변경 시점까지 계속 유효**하다. 즉, 2월이 1월과 동일하면 2월을 별도로 입력할 필요가 없고, 2월을 조회하면 1월 값이 그대로 반영된다.
- 3월부터 저축 비중을 변경한 경우, 3월에만 새 값을 입력하면 3월 이후는 변경된 값이 적용된다.

**예시 흐름**
```
저장된 레코드:      2026-01 (입력)       2026-03 (저축·투자 금액 변경)
                         |                        |
조회 결과:  2026-01 → 1월값
           2026-02 → 1월값 상속
           2026-03 → 3월값
           2026-04 → 3월값 상속
```

### 카테고리 (고정 Enum 8개)
| Enum | 라벨 |
|------|------|
| FOOD | 식비 |
| HOUSING | 주거 (집세/대출) |
| TRANSPORT | 교통 |
| EVENTS | 경조사 |
| COMMUNICATION | 통신 |
| LEISURE | 여가 |
| SAVINGS_INVESTMENT | 저축·투자 |
| ETC | 기타 |

### 핵심 계산식
- **저축율** = `SAVINGS_INVESTMENT 카테고리 금액 / 월급` (음수 허용, 초과 시 경고)
- **미사용/잔여** = `월급 - 카테고리 합계`
- **월간 추이 범위**: 최근 12개월 (상속 로직으로 각 월의 유효값을 계산)

## Why This Approach (변경 지점 기반 Effective Date 모델)

### 사용자 의도와의 부합
> "1월 입력 → 2월은 그대로 → 3월에 변경하면 1,2월은 기존 값, 3월부터 새 값으로 히스토리"

매월마다 동일한 값을 반복 입력하는 부담을 없애고, **실제 변경 시점만 기록**하여 UX와 데이터 효율성을 동시에 확보한다.

### 선택한 아키텍처
- **신규 도메인 모듈**: `salary` (기존 `portfolio`와 동일 DDD 계층 구조 적용)
- **Entity 2개** (정규화, JSON 컬럼 배제):
  - `MonthlyIncome` (id, userId, effectiveFromMonth, amount) — 월급 변경 이력
  - `SpendingConfig` (id, userId, category, effectiveFromMonth, amount, memo) — 카테고리별 지출 변경 이력
  - 유니크 제약: `(userId, effectiveFromMonth)` / `(userId, category, effectiveFromMonth)`
- **Enum**: `SpendingCategory` (8개 고정값)
- **조회 로직(상속)**: 특정 월 `M` 조회 시, 각 항목에 대해 `effectiveFromMonth <= M` 인 레코드 중 `MAX(effectiveFromMonth)` 를 사용
- **API**:
  - `GET /api/salary/monthly/{yearMonth}?userId=...` — 해당 월의 유효 값(상속 포함) 반환 (도넛+바 차트용)
  - `GET /api/salary/trend?userId=...&months=12` — 최근 N개월 각각의 유효 값 배열 반환 (라인 차트용)
  - `PUT /api/salary/income/{yearMonth}` — 해당 월을 기점으로 월급 값 upsert
  - `PUT /api/salary/spending/{yearMonth}/{category}` — 해당 월을 기점으로 카테고리 값 upsert
  - `DELETE /api/salary/income/{yearMonth}` — 해당 월의 월급 변경 레코드 제거 (이전 설정으로 돌아감)
  - `DELETE /api/salary/spending/{yearMonth}/{category}` — 해당 월의 카테고리 변경 레코드 제거 (이전 설정으로 돌아감)
  - `GET /api/salary/months?userId=...` — 변경 레코드가 존재하는 월 목록 (드롭다운/추이 기준점)

### 근거
1. **사용자 의도 일치**: 태형님이 직접 설명한 멘탈 모델(변경 지점만 기록, 나머지는 자동 상속)에 정확히 부합
2. **UX 우수**: 고정성 지출(주거, 통신 등)을 매월 재입력할 필요 없음
3. **저장 효율**: 변경이 없는 달은 레코드가 없어 저장 공간 절약
4. **정규화 유지**: JSON 컬럼 없이 2개 테이블로 관계형 설계
5. **portfolio 도메인 패턴 일관성**: DDD 계층(domain/application/infrastructure/presentation) 구조 유지

### 트레이드오프와 대응
- **일회성 변동 표현**: YAGNI로 이번 범위에서 제외. 필요하면 향후 override 레코드 + 다음 달 원복 레코드 조합을 위한 UI를 별도 설계
- **조회 로직 복잡도**: 상속 로직이 필요하지만, `MAX(effectiveFromMonth)` 서브쿼리로 표준 SQL로 처리 가능
- **중복 레코드 방지**: 서버에서 상속값과 입력값이 동일한 경우 실제 레코드를 생성하지 않고 클라이언트에 안내 응답을 반환하여 데이터 오염 방지

## Key Decisions

| 결정 사항 | 내용 |
|-----------|------|
| **데이터 모델** | **변경 지점 기반(Effective Date)** — 바뀐 달만 레코드 저장, 조회 시 이전 값 상속 |
| **카테고리** | 고정 Enum 8개 + 사용 내역별 메모 필드 |
| **입력 단위** | 카테고리당 1건 (합계 금액 입력, 거래 단위 아님) |
| **잔액 처리** | 도넛 차트에 "미사용/잔여" 세그먼트로 표시 (자동 계산) |
| **추이 차트** | 총 지출 / 월급 / 저축율 3개 라인 (12개월, 상속 로직 적용) |
| **추이 기간** | 최근 12개월 (오늘 기준 과거 12개월) |
| **저축율 공식** | 저축·투자 카테고리 / 월급 |
| **프론트엔드** | 기존 Alpine.js + Chart.js 패턴 유지 |
| **신규 탭** | `app.js` menus 배열에 `salary` 추가, HTML `x-show="currentPage === 'salary'"` |
| **userId 범위** | 기존 시스템과 동일 (userId 파라미터 필수) |
| **삭제 정책** | DELETE API 제공 (해당 월의 변경 레코드를 제거하여 이전 값으로 되돌림) |
| **초과 사용 표시** | 경고 색상(빨강 계열) + 음수 저축율 실제 값 표시 |
| **기본 월** | 탭 진입 시 마지막 변경 레코드가 존재하는 월을 자동 조회해서 표시 |
| **금액 데이터 타입** | `BigDecimal` (portfolio 도메인과 일관성, 정밀 결산) |
| **월 선택 UI** | 화살표 버튼(← →) + 드롭다운 겸용 (드롭다운에는 **변경 레코드가 있는 월만 노출**) |
| **입력 UX** | 상단 "입력 섹션" (월급 + 카테고리별 금액/메모 폼) + 하단 "차트 섹션" (도넛/바/라인) 분리 구성 |
| **빈 상태 UI** | 안내 메시지 ("월 실수령액을 먼저 입력해주세요") + 입력 유도 버튼 (온보딩 방식) |
| **금액 검증** | 0 이상 정수/실수 (음수 불가). 0은 허용 (쓰지 않은 카테고리 표현) |
| **새 월 추가 경로** | "새 월 기록하기" 버튼으로 **현재 월 이하**의 임의 월을 선택해 변경 레코드를 생성 (미래 월 선택 불가) |
| **추이 차트 X축** | 데이터 있는 범위만 표시 (기록 시작 이전 월은 차트에 포함하지 않음) |
| **중복 레코드 방지** | 상속된 값과 동일한 값으로 저장 시도 시 서버에서 비교 감지 → `409 Conflict` 또는 `200 + noop 메시지`로 안내. 불필요한 레코드 생성 차단 |
| **일회성 변동** | 이번 범위에서 제외 (YAGNI). 순수 변경 지점 모델만 지원 |

## Resolved Questions

브레인스토밍 과정에서 확정된 항목이다.

1. **데이터 모델** → **변경 지점 기반 Effective Date**
   - 사용자 의도: "바뀔 때만 입력, 나머지는 이전 값 상속"
2. **userId 범위** → **기존 시스템과 동일, userId 필수**
3. **삭제 정책** → **DELETE API 제공**
   - 삭제 의미: 해당 월의 변경 레코드 제거 → 이전 시점 설정이 다시 상속 적용됨
4. **초과 사용 처리** → **경고 색상 + 음수 저축율 실제 값 표시**
5. **진입 시 기본 월** → **마지막 변경 레코드가 있는 월**
   - 변경 레코드가 전혀 없으면 현재 월 빈 상태로 fallback
6. **월 선택 UI** → **화살표 + 드롭다운 (드롭다운은 변경 레코드 있는 월만)**
7. **입력 UX** → **입력 섹션 + 차트 섹션 분리**
8. **금액 타입** → **BigDecimal**
9. **빈 상태** → **안내 메시지 + 입력 유도 버튼**
10. **금액 검증** → **0 이상 (음수 불가)**
11. **새 월 이동** → **드롭다운에는 변경 레코드 있는 월만 노출, "새 월 기록" 버튼으로 추가 (현재 월 이하만)**
12. **일회성 변동 처리** → **이번 범위에서 제외 (YAGNI)** — 순수 변경 지점 모델만 지원
13. **추이 차트 빈 월 처리** → **데이터 있는 범위만 표시** (기록 시작 이전 월은 차트 X축에 포함하지 않음)
14. **중복 레코드 방지** → **서버에서 상속값과 비교 감지**, 동일 값일 경우 레코드 생성하지 않고 안내 메시지

## Out of Scope (YAGNI)

현재 범위에 포함하지 않는다.
- 카테고리 커스터마이징 (사용자 정의 카테고리)
- 거래 단위 기록 (카테고리당 다중 내역)
- 예산 설정 및 초과 알림
- 데이터 내보내기/가져오기 (CSV, Excel)
- 카테고리별 상세 추이 차트 (현재는 총합 라인만)
- 다중 수입원 (월급 외 부수입)
- 일회성 변동(override) 지원 — 향후 별도 설계 필요 시 추가
- 미래 월 선행 입력 (예: "5월부터 월세 인상 예정" 미리 기록)
- 추이 차트 빈 월 자동 0 채우기 (기록된 범위만 표시)
